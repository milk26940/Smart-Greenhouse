# 农业大棚监控网页制作说明

## 1. 目标

这份网页的目标是：

- 在浏览器中实时显示 STM32 发布到 MQTT 的温度数据
- 页面风格为农业大棚监控驾驶舱
- 不依赖后端服务，直接由浏览器连接 Mosquitto 测试服务器
- 首版只显示单温度点，不做历史曲线和控制功能

当前网页文件位于：

- `D:\STM32Project\Lora\网页\index.html`

---

## 2. 数据来源

STM32 接收端发布的 MQTT 数据如下：

- Broker 主机：`test.mosquitto.org`
- STM32 MQTT TCP 端口：`1883`
- 网页 MQTT WebSocket 端口：`8081`
- 主题：`lora/temperature`
- 消息格式：

```json
{"temperature":25.3,"source":"lora_rx"}
```

说明：

- STM32 固件继续使用普通 MQTT over TCP，所以走 `1883`
- 网页不能使用 `mqtt://`，只能使用 `ws://` 或 `wss://`
- 浏览器环境里最终采用的是安全 WebSocket：

```text
wss://test.mosquitto.org:8081/mqtt
```

---

## 3. 为什么网页一开始连不上

最开始网页连接的是明文 WebSocket：

- `ws://test.mosquitto.org:8080/mqtt`

调试日志显示：

- 原生 WebSocket 在浏览器中直接报错
- 关闭码是 `1006`
- `mqtt.js` 底层 stream 也报 `WebSocket error`

这说明问题不在 MQTT 主题，也不在 JSON，而是在浏览器对明文 WebSocket 的连接层。

后来把网页首选入口改成：

```text
wss://test.mosquitto.org:8081/mqtt
```

之后浏览器调试日志出现：

- 原生 WS 打开
- 收到 MQTT 包 `connack`
- 发送 `subscribe`
- 收到 `suback`

说明网页连接链路已经打通。

---

## 4. 页面结构设计

首版页面采用单文件静态页，包含这些区域：

1. 页面标题与本地时间
2. 当前温度主卡
3. MQTT 状态卡
4. 数据信息卡
5. 最近一条原始消息
6. 调试日志面板

页面风格不是普通后台表格，而是偏“农业驾驶舱”：

- 浅绿色和土黄色的背景层次
- 大号温度数字作为主视觉
- 卡片化布局
- 轻量动画和透明面板效果

---

## 5. 用到的技术

为了尽量简单，首版没有引入前端框架，也没有构建工具。

技术选择如下：

- HTML + CSS + 原生 JavaScript
- `mqtt.js` 通过 CDN 引入
- 浏览器直接连接 Mosquitto WebSocket

页面中引入 `mqtt.js` 的方式：

```html
<script src="https://unpkg.com/mqtt/dist/mqtt.min.js"></script>
```

## 6. 制作步骤

### 第一步：明确网页端和固件端的连接方式

先区分两个端口：

- 固件：`mqtt://test.mosquitto.org:1883`
- 网页：`wss://test.mosquitto.org:8081/mqtt`

这一点必须区分清楚，不能把 STM32 的 `1883` 直接搬到网页上。

### 第二步：确定页面只展示当前温度

为了尽快验证链路，首版只做：

- 当前温度
- 连接状态
- 主题与来源
- 最近更新时间
- 原始消息
- 调试日志

不做：

- 历史曲线
- 控制按钮
- 告警逻辑
- 后端存储

### 第三步：先写网页回归检查

在正式写页面之前，先写检查脚本，保证页面里必须存在：

- WebSocket 端点配置
- 主题配置
- 状态文案
- 调试日志区域

本次用到的检查脚本有：

- `D:\STM32Project\Lora\tests\check-greenhouse-dashboard.ps1`
- `D:\STM32Project\Lora\tests\check-greenhouse-broker-endpoints.ps1`
- `D:\STM32Project\Lora\tests\check-greenhouse-debug-log.ps1`

### 第四步：编写 `index.html`

网页文件放在：

- `D:\STM32Project\Lora\网页\index.html`

页面中定义了统一配置：

- 页面标题
- WebSocket 候选端点
- MQTT 主题
- 重连间隔

其中端点顺序为：

1. `wss://test.mosquitto.org:8081/mqtt`
2. `ws://test.mosquitto.org:8080/mqtt`
3. `ws://test.mosquitto.org:8080`
4. `ws://test.mosquitto.org/mqtt`

### 第五步：实现页面状态机

页面中明确区分以下状态：

- `连接中`
- `已连接`
- `已断开`
- `重连中`
- `数据异常`

页面加载后会自动：

1. 连接 broker
2. 订阅 `lora/temperature`
3. 收到消息后解析 JSON
4. 更新温度、来源、更新时间和原始消息

### 第六步：加入调试日志

为了定位网页为什么没数据显示，又在页面中加入了可见调试日志区。

日志会记录：

- 连接哪个端点
- 原生 WebSocket 是否打开
- `mqtt.js` 是否收到 `connack`
- 是否发送 `subscribe`
- 是否收到 `suback`
- 是否收到原始消息
- 是否 JSON 解析失败

这一步对排障非常重要。

## 7. 如何验证网页是否正常

### 方式一：直接看页面状态

正常时应该看到：

- MQTT 状态不是“连接中”
- 调试日志出现 `connack`
- 调试日志出现 `suback`
- 页面收到新的温度值

### 方式二：看调试日志

正常连接日志应类似：

```text
[02:04:49] 切换端点 -> wss://test.mosquitto.org:8081/mqtt
[02:04:49] 开始连接 -> wss://test.mosquitto.org:8081/mqtt
[02:04:51] 原生 WS 打开 -> wss://test.mosquitto.org:8081/mqtt
[02:04:51] 收到 MQTT 包 -> connack
[02:04:51] 连接成功 -> wss://test.mosquitto.org:8081/mqtt
[02:04:51] 发送 MQTT 包 -> subscribe
[02:04:52] 收到 MQTT 包 -> suback
[02:04:52] 订阅成功 -> lora/temperature
```

### 方式三：让 STM32 再发一条温度

网页只会显示“订阅成功之后收到的新消息”，不会自动取回历史旧消息。

所以订阅成功后，要让 STM32 再发一条温度，网页才会刷新。

正常时日志会继续出现：

```text
收到原始消息 -> {"temperature":26.2,"source":"lora_rx"}
收到消息 -> temperature=26.2, source=lora_rx
```

## 9. 后续可以继续扩展的方向

后面如果要继续做成真正的“智慧农业大棚系统”，可以扩展：

- 增加湿度、光照、土壤湿度等指标
- 增加历史曲线
- 增加告警阈值显示
- 增加多大棚切换
- 增加风机、水泵等控制区
- 增加后端服务和数据库

但建议顺序是：

1. 先保证单温度点实时显示稳定
2. 再做多指标
3. 最后再做控制和历史

---

## 10. 总结

这个网页的核心思路是：

- STM32 继续向公网 MQTT broker 发布温度
- 浏览器不经过后端，直接通过 `wss://test.mosquitto.org:8081/mqtt` 订阅同一主题
- 页面实时更新温度和连接状态
- 通过页面内调试日志快速定位浏览器侧问题

这套方式适合当前阶段做联调、演示和验证。
